# Дизайн `spec-v0.1` для формата IDF

**Дата:** 2026-04-19
**Статус:** дизайн-документ (брэйнсторм). Согласован с автором; следующий шаг — implementation plan через `writing-plans`.
**Источник:** `source/manifesto-v2.snapshot.md` (SHA-256 в `source/manifesto-v2.snapshot.sha256`).

## 1. Мотивация: почему спека до второй реализации

Манифест v2 §26 описывает направление развития «второй reference implementation» (Vue / Svelte / native mobile / Phoenix LiveView) как структурный стресс-тест на decoupling формата от React-specifics. В исходном плане он стоял **перед** написанием нормативной спеки.

Это методологически неверно. Без нормативной спеки independent implementer вынужден читать код референсной реализации как эталон. Получается **зеркало**, а не независимая реализация. Drift между манифестом и React-specifics не отлавливается, а наследуется. Тезис «формат уровня OpenAPI/JSON-LD» проверить невозможно — потому что у него нет аналога OpenAPI Specification.

Манифест v2 §3 сам признаёт: *«спека существует на уровне архитектурного решения и набросков, но не как написанный артефакт»*. §26 фиксирует её как **отдельный проект**, не минорную версию манифеста.

**Решение:** `spec-v0.1` — самостоятельный артефакт, развивающийся в репозитории `idf-spec/` независимо от `idf/` и `idf-sdk/`. Только после него — вторая реализация, с критерием успеха «implementer не открывал ни одного файла исходников референсной реализации».

## 2. Scope v0.1 (minimal viable normative)

### В скоупе нормативно

- **L1** — парсер онтологии, append-only Φ, `fold(Φ) → world`, `filterWorldForRole(world, viewer, ontology) → viewerWorld` с 4-приоритетным row-filter
- **L2** — `crystallize(intents, ontology, projection, [], {}) → artifact` с 6-фазным pipeline; 7 архетипов как enum + структуры слотов; `mergeProjections` для авторских override'ов
- **4 аксиомы формата** (§23 манифеста): детерминизм кристаллизации, viewer-scoping как тип данных, irreversibility как свойство эффекта, audit trail через Φ — в форме MUST
- **JSON Schema** для пяти core-объектов: `ontology`, `intent`, `effect`, `projection`, `artifact`
- **Test fixtures** на синтетическом эталонном домене (см. §5) — полные input → expected output для всех L1+L2 операций

### Вне скоупа v0.1 (Reserved for v0.2+)

Каждый пункт в спеке упомянут как `Reserved for v0.2+` с placeholder-секцией, но без нормативного содержания:

- **L3** — материализации pixel / voice / agent API / document
- **L4** — Pattern Bank `structure.apply` (фаза 5 кристаллизации); темпоральный scheduler; integrity-правило irreversibility в валидаторе; пять видов invariant'ов с handler'ами
- **Pattern Bank** целиком — даже matching-only паттерны
- **Base-таксономия ролей** (`owner | viewer | agent | observer`), `role.scope` для m2m, `preapproval` guard для agent
- **Семантические `fieldRoles`** (`money` / `percentage` / `coordinate` / ...) — упоминаются в спеке как opaque-string hint-механизм, без нормативного списка значений
- **Реактивные правила** (`ontology.rules`) с расширениями aggregation / threshold / schedule / condition
- **`shape`-слой** артефакта (`timeline` / `directory` / `default`) — производная от L4 Pattern Bank

### Обоснование scope-границы

L1+L2 + один эталонный домен в fixtures — минимум, который позволяет independent implementer:

1. Реализовать parser + fold + filterWorldForRole + crystallize
2. Проверить себя test-vectors-ами без чтения кода прототипа
3. Структурно столкнуться с тем же набором решений (как организовать Φ, как организовать crystallize pipeline, как реализовать filterWorldForRole row-filter), что и автор первой реализации — но без подсказок

L3 и L4 нужны для production, но **не нужны** для валидации тезиса «формат decoupled от React». Включение их в v0.1 — риск преждевременной фиксации того, что в манифесте описано как aspirational или roadmap-материал.

├── 01-conformance.md         # классы L1, L2; матрица обязательных функций;

│   ├── effect.schema.json
│   ├── projection.schema.json
│   └── artifact.schema.json
└── fixtures/
    └── library/              # синтетический эталонный домен (см. §5)
        ├── README.md         # описание домена, ролей, проекций
        ├── ontology.json
        ├── intents.json
        ├── projections.json
        ├── phi/              # input: списки confirmed effects по сценариям
        │   ├── empty.json
        │   ├── add-three-books.json
        │   ├── borrow-and-return.json
        │   └── ...
        ├── expected/

### Решения по конвенциям

- **JSON Schema draft-07** — стабильный, поддерживается всеми основными language ecosystems (Go, Rust, Python, Java, JS), tooling зрелый. Не draft-2020-12 (новее, патчи догоняют). Не OpenAPI — IDF не RPC.
- **Test fixtures — чистый JSON, без комментариев.** Implementer должен запускать через любой стандартный парсер без custom-pre-processing. Документация к fixture — в `README.md` рядом с ним.
- **Crystallize фаза 4–5 описана как hook-points в L2.** Сигнатура `(slots, context, ...) → slots` фиксирована, реализация L2 обязана прогонять noop-функцию в этих местах. Это сохраняет правильную структуру pipeline для будущего L4 без нарушения детерминизма L2.
- **Markdown файлы** — < 500 строк; разрастание → разбить на подсекции одного `.md` или вынести в подпапку.
- **Имена** — `lower-camelCase` для JSON-полей, `kebab-case` для имён файлов спеки.

## 4. Граница нормативного / информативного

### Нормативно (MUST)

- **Структуры данных:** JSON Schema для пяти core-объектов. Расширения через дополнительные поля разрешены (extensibility), обязательные поля обязаны присутствовать.
- **Интерфейсные функции (как контракты):** `fold`, `filterWorldForRole`, `crystallize` — заявленные сигнатуры, заявленные пред- и пост-условия, заявленный детерминизм. Реализация (язык, паттерн, transient state) — свободна.
- **Effects API:** life-cycle `proposed → confirmed | rejected`; append-only Φ; validator проверяется до confirm; rejected эффекты сохраняются отдельно.
- **4 аксиомы §23.**
- **Test fixtures.** Implementer считается conformant для L1+L2, если воспроизводит fixture-output на fixture-input.

### Информативно (SHOULD / MAY / описание)

- **Host-архитектура:** transport (HTTP / WebSocket / IPC / in-process); persistence (SQLite / Postgres / in-memory / append-only files); runtime (Node / Go / BEAM / browser-only). Спека описывает **интерфейсы**, не реализацию.
- **UI-адаптер и pixel-материализация:** в скоупе v0.1 только `artifact` как тип данных; pixel-рендеринг — L3, out of scope.
- **Validation tooling:** конкретный JSON Schema validator, конкретная test-runner среда — выбор implementer'а.
- **CLI / authoring environment** (Studio): не часть формата (манифест §21).

### Критерий conformance L1+L2

Implementer пишет: `parse(input) + apply L1+L2 functions + emit output`. Прогоняет через fixtures. Pass/fail — единственный critical path. Архитектурные решения (организация кода, язык, transport, persistence) — out of scope.

## 5. Эталонный домен `library`

### Почему синтетический, а не один из 9 production-доменов прототипа

Любой production-домен прототипа спроектирован под L4: содержит invariants, rules, иногда scheduler, иногда irreversibility. Использовать его как fixture для L1+L2 значит либо тащить L4-сложность в v0.1, либо arbitrary-усекать его — оба варианта дают шум. Implementer, читая такой fixture, будет вынужден угадывать, что является нормативной частью v0.1, а что — побочным эффектом производственной полноты.

Все нормативные спеки (HTTP RFC, OpenAPI, JSON Schema) используют **синтетические минимальные дидактические примеры**, не production-системы. Это та же стратегия.

### Описание домена

**Сущности (3):**

| Сущность | `kind` | Назначение |
|---|---|---|
| `User` | internal (owned) | Пользователь библиотеки |
| `Book` | **reference** | Общий каталог книг (видна всем без owner-проверки) |
| `Loan` | internal (owned через `userField`) | Запись о займе книги пользователем; имеет state machine `active → returned` |

**Роли (2):**

| Роль | Права |
|---|---|
| `reader` | owner для своих `Loan`; viewer для `Book` (через `Book.kind = "reference"`); не видит чужих `Loan` |
| `librarian` | admin: добавляет / удаляет `Book`; видит все `Loan` через расширенный `visibleFields` |

**Намерения (~7):**

| Intent | Owner | Effects |
|---|---|---|
| `add_book` | `librarian` | `α: create, entity: Book` |
| `remove_book` | `librarian` | `α: remove, entity: Book` |
| `borrow_book` | `reader` | `α: create, entity: Loan` (status `active`, userField → инициатор) |
| `return_book` | `reader` | `α: transition, entity: Loan` (`active → returned`) |
| `register_user` | `librarian` | `α: create, entity: User` |
| `update_book_metadata` | `librarian` | `α: replace, entity: Book` |
| `cancel_loan` | `librarian` | `α: remove, entity: Loan` |

**Проекции (5):** покрывают 4 архетипа из 7:

| Projection | Archetype | Доступна |
|---|---|---|
| `book-catalog` | `catalog` | reader + librarian |
| `book-detail` | `detail` | reader + librarian |
| `borrow-form` | `form` | reader |
| `my-loans` | `catalog` (viewer-scoped) | reader |
| `librarian-dashboard` | `dashboard` | librarian |

### Что покрывает домен

- **Четыре из пяти видов α активно**: create (`add_book`, `borrow_book`, `register_user`); replace (`update_book_metadata`); remove (`remove_book`, `cancel_loan`); transition (`return_book`). Пятый — `commit` — формально нормирован в спеке (зарезервирован для wizard-архетипа), но не покрыт fixture-вектором; см. Q-D3.
- **Три значения `entity.kind`**: internal-owned (`User`, `Loan`); reference (`Book`); ownership через `userField` (`Loan.userField → User.id`)
- **Чистый transition test**: `Loan.status: active → returned`
- **Две роли** с разными правами на одни данные → стресс-тест `filterWorldForRole`
- **Четыре из семи архетипов** в пяти проекциях (`catalog`, `detail`, `form`, `dashboard`) → покрытие L2 crystallize по разнообразию артефактов. Архетипы `feed`, `canvas`, `wizard` нормированы в спеке, но lightly-tested без fixture-вектора; см. Q-D3.
- **`mergeProjections`**: `borrow-form` имеет slot-override (явно зафиксированное поле `Loan.userField` → текущий viewer); тест auto-derived vs authored

### Что НЕ покрывает (и это сознательно)

- Pattern Bank (Reserved L4)
- Invariants с handler'ами (Reserved L4)
- Scheduler / темпоральные интенты (Reserved L4)
- m2m через `role.scope` (Reserved L4)
- voice / agent / document материализации (Reserved L3)

## 6. Процесс работы со спекой

### Принцип: спека не вводит новых аксиом

Если при написании спеки обнаруживается, что манифест не даёт однозначного ответа на нормативный вопрос, спека **не выдумывает**. Три действия:

1. **Минимальная нормативная позиция.** Беру самый узкий интерпретационный закрытый ответ, не противоречащий манифесту. Conservative-минимум. Например: «парсер L1 обязан принимать поле `fieldRole` как opaque строку; нормативного списка значений в v0.1 нет».

2. **Open question fixated.** В конце каждого `.md` файла спеки — секция `## Open questions` с записями:
   ```
   ### Q-XX: <короткое название>
   **Манифест говорит:** <цитата §N с номером>
   **Ambiguity:** <в чём именно непонятность>
   **v0.1 нормативная позиция:** <conservative-минимум>
   **Reserved for resolution in:** v0.2 / манифест v2.1 / TBD
   ```

3. **Обратная связь в манифест — отдельный артефакт.** В `idf-spec/feedback/manifesto-v2.md` собирается backlog мест, где манифест нужно расширить или уточнить. Это **результат** работы над спекой, а не её часть. Манифест на ходу из спеки не правится.

### Чего не делать

- Не открывать исходники прототипа (`idf/src/**`, `idf/server/**`) или SDK (`idf-sdk/packages/*/src/**`) для разрешения ambiguity. Соблазн «посмотреть, как они это сделали» — сигнал зафиксировать ambiguity как Open question, не разрешить угадыванием. Это и есть критерий честности эксперимента.
- Не использовать имплементационные числа из памяти контекста (имена файлов, версии пакетов, конкретные функции, числа тестов). Спека описывает формат, а не одну из его реализаций.

## 7. Изоляционная политика

Полный текст — в `idf-spec/CLAUDE.md`. Кратко:

**Allowed reads (white-list):**
- `idf-spec/source/manifesto-v2.snapshot.md` — единственный нормативный source
- `idf/docs/manifesto-v2.md` — допустимо для сверки, что snapshot не устарел
- `idf/docs/field-test-*.md` — полевые тесты как **примеры доменов на уровне дизайна** (не как образцы реализации)

**Forbidden reads (black-list):**
- `idf/src/**`, `idf/server/**`, `idf/scripts/**` — исходники прототипа
- `idf-sdk/packages/*/src/**` — исходники SDK-пакетов
- любые `*.test.{js,jsx,ts,tsx,cjs}` в обоих репо
- `idf/docs/implementation-status.md` — снимок реализации, не формата
- `idf/docs/archive/**` — архивные манифесты v1.x; нормативный source — только v2 snapshot
- `idf/docs/superpowers/specs/**` — внутренние спеки реализации
- `package.json`, `vite.config.js`, `vitest.config.js` соседних репо

**Правило сомнения:** если непонятно, относится ли файл к white или black-list — не читать, зафиксировать как открытый вопрос.

## 8. Критерий успеха `spec-v0.1`

Independent implementer (человек или команда), получив доступ только к содержимому `idf-spec/` (без `idf/` и `idf-sdk/`):

1. **Понимает**, какие пять core-объектов нужно поддерживать (через JSON Schema)
2. **Реализует** L1: парсер онтологии, append-only Φ, `fold`, `filterWorldForRole`
3. **Реализует** L2: 6-фазный crystallize, 7 архетипов, `mergeProjections`
4. **Прогоняет** fixtures домена `library` и видит pass/fail на каждом expected output
5. **Проходит все тесты** без обращений к референсной реализации

Если этот сценарий выполняется — `spec-v0.1` достигла цели. Если на каком-то шаге implementer вынужден заглянуть в код референсной реализации — это сигнал, что спека неполна или неоднозначна; ambiguity фиксируется в Open questions, спека дорабатывается, цикл повторяется.

## 9. Open questions для самого дизайна (не для спеки)

Места, где сам дизайн ещё не зафиксирован, и которые надо решить при переходе к writing-plans или в первой итерации спеки:

- **Q-D1: Формат сравнения artifact в fixtures.** Семантическая равенство (deep-equal с игнорированием порядка ключей) или строго-байтовое? Рекомендация: семантическое; реализатор сам выбирает comparator. Зафиксировать в `00-introduction.md`.
- **Q-D2: Версионирование fixture-формата.** Если спека эволюционирует, fixtures могут устареть. Решение: fixtures versioned вместе со спекой, manifest snapshot SHA-256 указан в каждом fixture-файле как `_meta.specVersion` и `_meta.manifestSha256`. **Принять сейчас.**
- **Q-D3: Что делать с архетипами `feed` / `canvas` / `wizard` и эффектом `α: commit`, если они не покрыты `library`-доменом?** Минимально нормировать в спеке (структура слотов / lifecycle), без fixture-вектора. Помечать как `Lightly-tested in v0.1`. Альтернатива — расширить `library` (например, добавить multi-step резерв книги через wizard, что даст commit) или добавить второй mini-domain в v0.1. **Рекомендация:** lightly-tested + примечание в `01-conformance.md`, что implementer SHOULD реализовать всю поверхность L2, но MUST проходит только fixture-векторы. Расширение fixture-покрытия — задача v0.2.
- **Q-D4: Имя для `α` поля в JSON.** Манифест использует греческую букву; в JSON это либо `"alpha"`, либо `"a"`, либо `"kind"`. Рекомендация: `"kind"` (читаемо, ASCII, согласуется с `entity.kind`). Зафиксировать в `effect.md` и `effect.schema.json`.
- **Q-D5: Где живёт `_meta` в fixtures?** Top-level или внутри отдельного `_meta` объекта? Рекомендация: top-level `_meta` объект в каждом fixture-файле, поля `specVersion`, `manifestSha256`, `description`.

Эти вопросы решаются в первой итерации спеки и фиксируются как нормативные.
