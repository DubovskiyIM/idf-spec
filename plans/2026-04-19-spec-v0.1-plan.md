> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Написать `spec-v0.1` — нормативную спецификацию формата IDF для conformance classes L1+L2 — в репозитории `idf-spec/` без чтения исходников референсной реализации.

**Architecture:** Спека состоит из (1) markdown-прозы в `spec/`, разбитой по объектам и алгебре; (2) JSON Schema draft-07 для пяти core-объектов в `spec/schemas/`; (3) fixture-векторов на синтетическом домене `library` в `spec/fixtures/library/`. Прозу пишет implementer по outline'ам в плане; JSON Schemas и одиночные fixture-примеры даны в плане целиком; остальные fixtures — по образцу первого.

**Tech Stack:**
- Markdown (CommonMark) для прозы
- JSON Schema draft-07 для нормативных структур
- Чистый JSON (без комментариев) для fixtures
- Node + AJV (`ajv-cli`) — self-validation tooling, не часть нормативного формата

**Source of truth:** `idf-spec/source/manifesto-v2.snapshot.md` (SHA-256 в `idf-spec/source/manifesto-v2.snapshot.sha256`). Любые ambiguities манифеста фиксируются как Open questions в спеке + в `idf-spec/feedback/manifesto-v2.md`. Чтение исходников `idf/src/`, `idf/server/`, `idf-sdk/packages/*/src/` запрещено политикой `idf-spec/CLAUDE.md`.

**Locked decisions (от design doc и брэйнсторма):**
- α-поле эффекта в JSON называется `kind` (Q-D4)
- Каждый fixture-файл имеет top-level `_meta: {specVersion, manifestSha256, description}` (Q-D2, Q-D5)
- Семантическое сравнение artifact в fixtures (deep-equal с игнорированием порядка ключей объектов; для массивов порядок важен) (Q-D1)
- `filterWorldForRole` в v0.1 — **3-приоритетный** row-filter (reference → ownerField → none); `role.scope` для m2m — Reserved L4
- Архетипы `feed`, `canvas`, `wizard` и α-эффект `commit` нормированы в спеке, но не покрыты fixture-вектором — `Lightly-tested in v0.1` (Q-D3)

### Task 2: Написать `validate.sh` runner

**Files:**
- Create: `validate.sh`

- [ ] **Step 1: Написать `validate.sh`**

```bash
#!/usr/bin/env bash
# Self-validation спеки: проверяет, что (1) JSON Schemas валидны как
# draft-07 и (2) fixtures валидируются против соответствующих schemas.
# Это infrastructure для авторов спеки, не часть нормативного формата.

set -euo pipefail

cd "$(dirname "$0")"

SCHEMAS_DIR="spec/schemas"
FIXTURES_DIR="spec/fixtures"

if [ ! -d "$SCHEMAS_DIR" ]; then
  echo "Нет $SCHEMAS_DIR — ничего валидировать."
  exit 0
fi

# Step 1: каждый файл в SCHEMAS_DIR должен быть корректным JSON Schema draft-07.
echo "== Schemas (draft-07 well-formedness) =="
for schema in "$SCHEMAS_DIR"/*.schema.json; do
  [ -e "$schema" ] || { echo "  (нет .schema.json в $SCHEMAS_DIR)"; break; }
  echo "  $schema"
  npx ajv compile -s "$schema" --strict=false > /dev/null
done

# Step 2: fixtures валидируются против схем по конвенции имени:
#   ontology.json     → ontology.schema.json
#   intents.json      → каждый элемент массива против intent.schema.json
#   projections.json  → каждый элемент массива против projection.schema.json
#   phi/*.json        → каждый элемент phi-массива против effect.schema.json
#   expected/world/*.json — не валидируются формально (это world-snapshot, нет схемы в v0.1)
#   expected/artifact/*.json → каждый файл против artifact.schema.json
#   expected/viewer-world/*.json — не валидируются формально

if [ -d "$FIXTURES_DIR" ]; then
  echo "== Fixtures =="
  for domain_dir in "$FIXTURES_DIR"/*/; do
    domain="$(basename "$domain_dir")"
    echo "  domain: $domain"

    [ -f "$domain_dir/ontology.json" ] && \
      npx ajv validate -s "$SCHEMAS_DIR/ontology.schema.json" \
                       -d "$domain_dir/ontology.json" --strict=false

    if [ -f "$domain_dir/intents.json" ]; then
      npx ajv validate -s "$SCHEMAS_DIR/intent.schema.json" \
                       -d "$domain_dir/intents.json#" --strict=false || true
      # ajv-cli не умеет per-element массива; для v0.1 проверяем top-level как массив объектов
      # через wrapper-схему — см. note ниже
    fi

    if [ -d "$domain_dir/phi" ]; then
      for phi in "$domain_dir/phi"/*.json; do
        [ -e "$phi" ] || break
        echo "    phi: $phi"
        # phi-файлы валидируются wrapper-схемой phi.schema.json (создаётся в task 6)
        npx ajv validate -s "$SCHEMAS_DIR/phi.schema.json" \
                         -d "$phi" --strict=false
      done
    fi

    if [ -d "$domain_dir/expected/artifact" ]; then
      for art in "$domain_dir/expected/artifact"/*.json; do
        [ -e "$art" ] || break
        echo "    artifact: $art"
        npx ajv validate -s "$SCHEMAS_DIR/artifact.schema.json" \
                         -d "$art" --strict=false
      done
    fi
  done
fi

echo "== OK =="
```

```markdown
# Effect

## Источник
Манифест v2 §7 (`source/manifesto-v2.snapshot.md`).

## Назначение
Эффект — атом изменения мира. Все изменения мира в IDF происходят через
confirmed effects, складываемые в append-only лог Φ. Мир не редактируется
in-place; он перевычисляется через `fold(Φ, ontology) → world` (см.
`04-algebra/fold.md`).

## Структура (нормативно)

JSON Schema: `spec/schemas/effect.schema.json`.

| Поле     | Тип    | Обязательно | Описание |
|----------|--------|-------------|----------|
| `kind`   | enum   | MUST        | Вид изменения: один из `create`, `replace`, `remove`, `transition`, `commit`. Соответствует символу α в манифесте §7. |
| `entity` | string | MUST        | Имя сущности из `ontology.entities`. |
| `fields` | object | SHOULD      | Полезная нагрузка эффекта; для `remove` минимум `{id: ...}`. |
| `context`| object | MUST        | Метаданные. v0.1 нормирует обязательное поле `context.at` (ISO 8601 timestamp). Остальные поля opaque. |

### Почему `kind` вместо `α`
Манифест использует греческую букву α; в JSON это либо `"alpha"`, либо
`"a"`, либо имя по существу. Нормативно выбрано `"kind"`: ASCII, читаемо,
согласуется с `entity.kind` в онтологии. Греческая буква α в манифесте
остаётся как референс на математическую традицию; в формате — `kind`.

### Пять видов изменения

(Перечисление каждого: `create` — добавить новую сущность; `replace` —
заменить поля существующей; `remove` — удалить; `transition` — изменить
состояние workflow; `commit` — зафиксировать завершение сложной частицы.
По одному параграфу на каждый, с примером.)

#### `commit` в v0.1
Эффект `commit` нормирован, но не покрыт fixture-вектором эталонного
домена (см. `01-conformance.md` Lightly-tested). Implementer SHOULD
поддерживать его в парсере.

## Lifecycle

`proposed → { confirmed | rejected }`.

(Параграф: эффект сначала имеет статус `proposed` — он находится в Δ
(черновики); validator проверяет инварианты и условия намерения; на
успехе → `confirmed`, попадает в Φ; на провале → `rejected`, не попадает
в Φ, но сохраняется в отдельной зоне для audit'а. Δ session-scoped:
исчезает при cancel или истечении сессии.)

В v0.1 нормируется только статус `confirmed`. Δ-storage (как хранить
proposed/rejected) — host-level concern, информативно.

## Упорядочивание Φ

Φ упорядочена по `context.at` (timestamp confirm'а). Если два эффекта
имеют одинаковый `at` (теоретически возможно), порядок определяется
позицией в массиве Φ — первый в массиве считается более ранним.

(Это нормирует tie-breaker; манифест не специфицирует, но для
детерминизма fold нужен явный rule. Помечено как Open question Q-1.)

## Validation

Validator MUST проверить эффект на:
- Валидность kind, entity (entity существует в ontology)
- Валидность context.at (ISO 8601)

Дополнительные проверки (invariant'ы 5 видов, irreversibility integrity)
— Reserved L4.

## Cross-references

- `spec/02-axioms.md` — аксиома 4 (audit trail через Φ)
- `spec/03-objects/intent.md` — intents эмитят эффекты при подтверждении
- `spec/03-objects/ontology.md` — entity.fields определяют допустимый shape effect.fields

### Q-1: Tie-breaker при одинаковых timestamps
**Манифест говорит:** §11 — «эффекты в Φ упорядочены по времени, и fold применяет их строго в этом порядке».

**v0.1 нормативная позиция:** порядок определяется позицией в массиве Φ — первый в массиве считается более ранним. Implementer MUST сохранять стабильный порядок при сериализации/десериализации Φ.
**Reserved for resolution in:** манифест v2.1 (явное правило ordering или требование sub-millisecond timestamps).

**Ambiguity:** v0.1 не нормирует `__irr` (Reserved L4).
**v0.1 нормативная позиция:** поле зарезервировано как opaque object; парсер MUST принимать его без ошибок, но не обязан интерпретировать.
**Reserved for resolution in:** v0.2+ (integrity-правило для irreversibility).
```

```markdown
# fold

## Источник
Манифест v2 §11.

## Сигнатура (нормативно)

```
fold(Φ, ontology) → world
```

- `Φ` — массив confirmed эффектов (см. `spec/03-objects/effect.md`),
  упорядоченный по `effect.context.at` ASC, tie-breaker — позиция в
  массиве.
- `ontology` — ontology-объект (см. `spec/03-objects/ontology.md`).
- `world` — JSON-объект `{EntityName: {entityId: entityRecord, ...}, ...}`.

## Аксиома детерминизма

`fold` — чистая функция. Один и тот же `(Φ, ontology)` MUST возвращать
identically equal world (deep-equal с игнорированием порядка ключей).

## Алгоритм (нормативно)

1. Инициализировать `world = {}`.
2. Для каждого entity name из `ontology.entities`: `world[entityName] = {}`.
3. Для каждого `effect` в Φ (в порядке `at` ASC, tie-breaker: позиция):
   - Применить `effect` к `world` согласно `effect.kind`:
     - `create`: `world[entity][fields.id] = fields`. Если `fields.id` уже
       присутствует — это ошибка fold'а (MUST). (Т.к. в v0.1 нет
       referential invariants — это единственная встроенная проверка.)
     - `replace`: `world[entity][fields.id] = {...world[entity][fields.id], ...fields}`.
       Если `fields.id` отсутствует — ошибка.
     - `remove`: удалить `world[entity][fields.id]`. Если отсутствует —
       no-op (MAY также выдать warning, но не ошибку).
     - `transition`: `world[entity][fields.id] = {...existing, ...fields}`.
       (В L1+L2 не отличается от replace; различие в L4 — invariant'ы
       transition kind проверяют допустимость перехода.)
     - `commit`: no-op (Lightly-tested, Reserved для wizard в L4+).
4. Вернуть `world`.

## Производные функции

`foldFiltered(Φ, predicate)` и `foldView(Φ, viewer, ontology)` —
manifest §11; в v0.1 определены информативно (см. cross-references), не
обязательны для L1.

**Reserved for resolution in:** v0.2+ (если invariant'ы потребуют strict semantics).
```

```markdown
# filterWorldForRole

## Источник
Манифест v2 §14.

## Сигнатура (нормативно)

```
filterWorldForRole(world, viewer, ontology) → viewerWorld
```

- `world` — output `fold(Φ, ontology)` (см. `04-algebra/fold.md`).
- `viewer` — `{role: <roleName>, id: <userId>}`. `id` — id viewer-сущности (используется для row-filter по ownerField).
- `ontology` — определяет правила scoping.
- `viewerWorld` — `{EntityName: {entityId: entityRecord, ...}}`, тот же тип что world, но содержит только видимые сущности с видимыми полями.

## Row-filter (3-приоритетный, нормативно)

Для каждой entity в `ontology.entities`:

1. **Если entity не упомянута в `role.visibleFields`** — entity не входит в viewerWorld (даже как пустой namespace).
2. **Если `entity.kind == 'reference'`** — все записи entity видны (без owner-проверки). Применить column-filter (visibleFields).
3. **Иначе если `entity.ownerField` задан** — видны записи, где `record[ownerField] === viewer.id`. Применить column-filter.
4. **Иначе** — none. Entity видна структурно, но row-filter не пропускает ни одну запись (privacy by default).

## Column-filter

Для каждой видимой записи: оставить только поля из `role.visibleFields[Entity]`.
Если `visibleFields[Entity] === '*'` — все поля.
Иначе — пересечение.

`id`-поле (или поле, помеченное как `ownerField` для self-owned сущностей)
SHOULD всегда оставаться видимым — иначе viewerWorld становится
ambiguous-keyed. (Это implementer-рекомендация; формально в v0.1 не
enforce'ится.)

## Аксиома viewer-scoping (cross-ref аксиома 2)

`viewerWorld` — тип данных для конкретного viewer'а, не post-фильтр
рендера. Любая материализация читает `viewerWorld`, не `world`.

## Reserved L4

- Приоритет 0 (`role.scope` для m2m через assignment-bridge) — Reserved.
  В v0.1 row-filter — 3-приоритетный, не 4.
- Composite/polymorphic entities — Reserved.
```

```markdown
# Axioms

Четыре аксиомы формата (манифест v2 §23). Любая реализация, нарушающая
одну из них, не является конформной — вне зависимости от уровня
L1–L4.

## 1. Детерминизм кристаллизации

`crystallize(intents, ontology, projection, viewer, viewerWorld)` — чистая
функция. Два вызова с identically equal входом MUST возвращать
identically equal artifact (semantic equality).

Никаких скрытых состояний, временных зависимостей, обращений к сетевым
источникам, LLM-побочных эффектов.

**Cross-reference:** `04-algebra/crystallize.md`. Conformance проверяется
fixture-векторами `expected/artifact/` — каждый файл MUST воспроизводиться
из соответствующего `(intents, ontology, projection, viewer, viewerWorld)`
input'а.

## 2. Viewer-scoping — тип данных, не фильтр

`viewerWorld = filterWorldForRole(world, viewer, ontology)` — это **тип
данных** для конкретного viewer'а, не post-фильтр render'а.

Любая материализация (в v0.1 — только artifact-генерация через crystallize;
L3+ материализации — Reserved) MUST читать `viewerWorld`, не `world`.

Implementer L1+L2: MUST не передавать `world` в crystallize; MUST
передавать `viewerWorld`.

**Cross-reference:** `04-algebra/filter-world.md`. Conformance проверяется
fixture-векторами `expected/viewer-world/`.

## 3. Irreversibility — свойство эффекта, не процесса (Reserved L4)

В v0.1 эта аксиома **нормирована, но не enforce'ится** на L1+L2 уровне.

Полная формулировка (для v0.2+): эффект с
`context.__irr = {point: "high", at: <past ISO>}` MUST не отменяться
через `α: "remove"` на целевой сущности. `α: "replace"` (forward
correction) разрешён всегда.

В v0.1 implementer MUST принимать `__irr` как opaque поле без
интерпретации. Integrity-правило (Reserved L4) добавится в v0.2+.

**Cross-reference:** `03-objects/effect.md` (поле `context.__irr`).

## 4. Audit trail через Φ

Любое изменение мира MUST происходить через confirmed effect в Φ. Мир
не редактируется in-place (никаких `world[entity][field] = value`).

Это влечёт:
- `rebuild from Φ` всегда даёт текущее состояние мира
- Причинность сохраняется
- Audit-запросы («кто изменил это поле? когда? каким intent'ом?»)
  отвечаются запросом к Φ, не требуют отдельной audit-инфраструктуры

Implementer L1+L2: persistence-стратегия (как хранить Φ — SQLite,
Postgres, append-only files, in-memory) — host-level concern,
информативно. Нормативно: Φ MUST быть append-only; effects MUST не
mutate'иться после confirm.

**Cross-reference:** `03-objects/effect.md` (lifecycle), `04-algebra/fold.md`
(rebuild from Φ).
```
